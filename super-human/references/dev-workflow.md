# 开发工作流（来自 superpowers）

本文档定义开发模式的5个阶段工作流程。

---

## 阶段1: 澄清 (Clarify)

**触发**: 用户想要构建某物。在接触任何代码之前激活。

**检查清单**（按顺序）:

1. **探索项目上下文** — 检查文件、文档、最近提交
2. **问澄清问题** — 一次一个，优先多项选择题
3. **提出2-3个方法** — 带权衡和建议
4. **呈现设计** — 按复杂性分节，每节后获得批准
5. **写设计文档** — `docs/plans/YYYY-MM-DD-<topic>-design.md` → 提交
6. **过渡** — 调用 writing-plans 阶段

**规则**:

- 每条消息只问一个问题
- 优先使用多项选择题而非开放性问题
- 每个项目都要经过此流程 — 没有"简单"项目的例外
- **硬门**: 在设计获批之前，禁止编写代码、搭建脚手架或实现任何东西！
- 在稳定之前，提出2-3个方法，带头建议

**要问的问题**:

- 你真正想做什么？（目的）
- 存在什么限制？（时间、技术栈、依赖）
- 成功是什么样子的？（成功标准）
- 这个不应该做什么？（范围边界）

**设计部分要涵盖**:

- 架构概览
- 组件及其职责
- 数据流
- 错误处理方法
- 测试策略

**设计批准后**:

- 将设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 提交设计文档
- 移交给 writing-plans — 不使用其他技能，不实现

---

## 阶段2: 写计划 (Writing Plans)

**触发**: 设计获批。由头脑风暴阶段激活。

**摘要**:

- 写详细的逐任务实施计划
- 每个任务 = 2–5分钟：写测试 → 观察失败 → 实现 → 观察通过 → 提交
- 保存到 `docs/plans/YYYY-MM-DD-<feature>.md`
- 宣布：「我正在使用 writing-plans 技能创建实施计划。」
- 保存后，提供两种执行模式：
  - **子代理驱动（当前会话）**: 每个任务使用 `sessions_spawn` + 两阶段审查
  - **手动执行**: 用户自己运行任务

**计划模板**:

```markdown
# 实施计划：<功能名称>

## 概述
<简要描述>

## 任务列表

### 任务1: <任务名称>
- **目标**: <目标>
- **测试**: <测试描述>
- **实现**: <实现描述>
- **验证**: <验证步骤>

### 任务2: <任务名称>
...

## 风险
<风险列表>

## 依赖
<依赖列表>
```

---

## 阶段3: 子代理驱动开发 (Subagent-Driven Development)

**触发**: 计划存在，用户选择子代理驱动的执行。

**每任务循环（OpenClaw）**:

1. `sessions_spawn` 一个实现者子代理（带任务和完整计划上下文）
2. 等待完成通知
3. `sessions_spawn` 一个规范审查者子代理 → 必须确认代码符合规范
4. `sessions_spawn` 一个代码质量审查者子代理 → 必须批准质量
5. 修复任何问题，必要时重新审查
6. 标记任务完成，移动到下一个
7. 最终：调度整体代码审查者 → 移交给阶段5

**TDD是强制性的**。参见 [references/tdd.md](references/tdd.md).

**子代理调度模式**:

```
Goal: [一句话]
Context: [为什么重要，哪个计划文件]
Files: [精确路径]
Constraints: [不要做什么 — 无范围蔓延，仅TDD]
Verify: [如何确认成功 — 测试通过，特定命令]
Task text: [从计划中粘贴完整任务]
```

使用 `sessions_spawn` 运行带有任务的详细提示。子代理自动宣布结果。

## 阶段4: 系统化调试 (Systematic Debugging)

**触发**: 错误、测试失败、意外行为 — 任何技术问题。

**硬门**: 没有根因调查，禁止修复！

**四个阶段**:

1. **根因调查** (读取错误、重现、检查最近更改、追踪数据流)
2. **模式分析** (找到工作示例、比较、识别差异)
3. **假设+测试** (一次一个假设，测试证明/反驳)
4. **修复+验证** (在根因修复，而非症状；验证修复不会破坏任何东西)

**调试检查清单**:

- [ ] 已读取完整错误堆栈
- [ ] 已重现问题
- [ ] 已检查相关的最近更改
- [ ] 已追踪数据 Flow
- [ ] 已找到工作示例并比较
- [ ] 已制定假设并测试
- [ ] 已修复根因而非症状
- [ ] 已验证修复不会破坏任何东西

---

## 阶段5: 完成分支 (Finishing a Branch)

**触发**: 所有任务完成，所有测试通过。

**摘要**:

1. 验证所有测试通过
2. 确定基础分支
3. 提供4个选项：本地合并 / 推送+PR / 保持 / 丢弃
4. 执行选择
5. 清理

**选项详情**:

| 选项 | 描述 |
|------|------|
| 本地合并 | 合并到基础分支，不创建PR |
| 推送+PR | 推送到远程并创建Pull Request |
| 保持 | 保持分支不变 |
| 丢弃 | 丢弃分支，丢弃所有工作 |

**清理检查清单**:

- [ ] 所有测试通过
- [ ] 代码已审查
- [ ] 文档已更新
- [ ] 分支已合并或PR已创建
- [ ] 临时文件已删除
- [ ] 分支已清理（如需要）

---

## 关键原则

- **一次一个问题** 在头脑风暴期间
- **TDD始终** — 先写失败的测试，删除在测试前写的代码
- **YAGNI** — 从所有设计中删除不必要的功能
- **DRY** — 无重复
- **系统化优于临时** — 遵循流程，特别是在时间压力下
- **证据优于主张** — 在声明成功之前验证
- **频繁提交** — 每次测试通过后
