# 模式识别规则

本文档定义如何自动识别用户意图，选择合适的工作流模式。

---

## 模式分类

### 1. 开发模式 (Development Mode)

**触发词**:
- 「开发」、「写代码」、「做个XX」、「实现XX功能」
- 「帮我做」、「工作流」、「任务分解」
- 「构建」、「编码」、「编程」

**识别规则**:
1. 用户提到具体的功能、特性或应用
2. 需要编写代码或构建系统
3. 涉及技术栈、框架或工具选择

**加载的工作流**: superpowers 5-phase 流程
1. 澄清需求 (Clarify)
2. 制定设计 (Plan)
3. 执行开发 (Execute with TDD)
4. 系统调试 (Debug)
5. 完成分支 (Finish)

---

### 2. 商业管理模式 (Business Mode)

**触发词**:
- 「客户」、「项目」、「定价」、「提案」
- 「机构」、「代理机构」、「服务」
- 「交付物」、「团队协调」

**识别规则**:
1. 用户提到客户、项目或商业运营
2. 需要管理代理机构或自由职业业务
3. 涉及定价、提案或客户沟通

**加载的工作流**: agency 7-area 流程
1. 客户接入 (Client Onboarding)
2. 定价提案 (Pricing & Proposals)
3. 项目跟踪 (Project Tracking)
4. 交付物管理 (Deliverables)
5. 团队协调 (Team Coordination)
6. 学习系统 (Learning System)
7. 按类型定制 (Agency Type Specifics)

---

### 3. 设计模式 (Design Mode)

**触发词**:
- 「设计」、「UI」、「UX」、「海报」、「LOGO」
- 「视觉」、「图形」、「视频」、「品牌」
- 「审美」、「风格」、「色彩」

**识别规则**:
1. 用户提到视觉设计、UI/UX或品牌
2. 需要创建设计稿或视觉作品
3. 涉及审美偏好或风格选择

**加载的工作流**: design skill 自动学习流程
1. 检测视觉偏好
2. 生成符合品味的设计
3. 自适应调整

---

### 4. 专家模式 (Expert Mode)

**触发词**:
- 「作为XX专家」、「我需要XX师」、「用XX的角色」
- 「咨询」、「请教」、「指导」

**识别规则**:
1. 用户明确指定需要某个领域的专家
2. 需要专业建议或深度分析
3. 从193个专业角色中选择

**加载的工作流**: agency-agents-zh 角色加载流程
1. 识别所需专家类型
2. 从角色库加载对应定义
3. 按专家行为模式执行任务

---

## 混合模式识别

### 开发+设计

**示例**: "帮我设计一个博客网站并开发出来"

**处理策略**:
1. 先进入设计模式 → 创建设计稿
2. 再进入开发模式 → 实现设计
3. 使用design skill执行设计，superpowers流程实现

### 商业+专家

**示例**: "作为安全专家，帮我给客户做安全审计提案"

**处理策略**:
1. 先加载专家角色 → 安全工程师
2. 再进入商业管理模式 → 创建提案
3. 结合专业知识和商业流程

### 开发+专家

**示例**: "作为后端架构师，帮我设计这个系统的架构"

**处理策略**:
1. 先加载专家角色 → 后端架构师
2. 再进入开发模式 → 架构设计阶段
3. 按专家标准输出架构文档

## 模式切换规则

### 自动切换

**触发条件**:
- 用户的新的请求明显属于不同模式
- 当前任务完成，用户提出新类型的任务

**示例**:
```
用户：帮我做一个网站（开发模式）
AI：[完成网站开发]
用户：现在帮我给这个网站定个价（商业管理模式）
AI：[自动切换到商业管理模式]
```

### 手动切换

**触发词**:
- 「用开发模式」、「切换到开发模式」
- 「用商业模式」、「切换到商业模式」
- 「用设计模式」、「切换到设计模式」

**处理**: 忽略自动识别，强制使用指定模式

---

## 冲突解决

###  ambiguous请求

**示例**: "帮我做个东西"

**处理策略**:
1. 反问澄清：「你想做什么类型的东西？」
2. 提供选项：
   - 开发应用/网站 → 开发模式
   - 设计视觉作品 → 设计模式
   - 管理客户项目 → 商业管理模式
   - 咨询专业问题 → 专家模式

### 多重意图

**示例**: "帮我做个网站，要好看，还要考虑怎么卖"

**处理策略**:
1. 识别主要意图（做网站 → 开发模式）
2. 识别次要意图（要好看 → 设计模式；考虑怎么卖 → 商业管理模式）
3. 制定多阶段计划：
   - 阶段1: 设计模式（设计UI）
   - 阶段2: 开发模式（实现网站）
   - 阶段3: 商业管理模式（制定销售策略）

## 模式识别优先级

**高优先级**（立即切换模式）:
- 用户明确指定模式：「用开发模式」
- 触发词清晰且唯一：「帮我做个网站」→ 开发模式

**中优先级**（询问确认）:
- 触发词模糊：「帮我做个东西」→ 反问澄清
- 多重意图：「帮我做个网站，要好看」→ 提供分阶段计划

**低优先级**（保持当前模式）:
- 小的偏离：在开发模式中，用户问个简单的设计问题
- 上下文延续：明显的后续问题

## 记忆系统协同

### 模式相关的记忆存储

**开发模式记忆** → `~/memory/projects/`
**商业管理模式记忆** → `~/memory/clients/` + `~/agency/`
**设计模式记忆** → `~/memory/designs/`
**专家模式记忆** → `~/memory/knowledge/`

### 跨模式记忆共享

**项目连续性**:
- 开发项目可能来自商业模式的客户
- 设计作品可能用于开发项目
- 专家建议可能服务于商业决策

**策略**: 在所有模式中都维护 `~/memory/INDEX.md`，确保信息可跨模式检索

## 调试和日志

**缓存机制**:
- 对频繁使用的模式，缓存识别规则
- 对用户历史偏好，优先推荐常用模式

**学习机制**:
- 从用户纠正中学习
- 更新模式识别规则
- 优化冲突解决策略
